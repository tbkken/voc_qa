# ADR Writer

将技术讨论转化为结构化的架构决策记录，让未来的团队成员理解"为什么"。

## ADR 文件规范

- 存放位置：`docs/adr/`
- 命名格式：`ADR-{编号}-{短标题}.md`，如 `ADR-001-use-postgresql.md`
- 状态流转：草稿 → 提议 → 已接受 / 已废弃 / 已取代

## ADR 模板

```markdown
# ADR-{编号}：{决策标题}

**状态**：已接受
**日期**：YYYY-MM-DD
**决策者**：@xxx, @xxx
**关联需求**：#ticket-id

---

## 决策

[一句话说清楚做了什么决定。]
```

## 常见 ADR 场景示例

### 数据库选型

```markdown
# ADR-001：主数据库选用 PostgreSQL

**状态**：已接受
**日期**：2024-01-15

## 背景
项目启动，需要选择主数据库。核心数据包括用户信息、订单、商品，
均为结构化数据。部分场景（如商品属性）需要灵活的 JSON 存储。
团队有 MySQL 和 PostgreSQL 使用经验。

## 决策
使用 PostgreSQL 16 作为主数据库。

## 考虑的选项

### 选项 A：PostgreSQL（已选）
- 优点：JSON/JSONB 支持更完善；事务隔离级别更丰富；支持更多索引类型
- 缺点：团队 MySQL 经验更多，有学习成本

### 选项 B：MySQL 8
- 优点：团队熟悉，运维经验丰富
- 缺点：JSON 支持较弱；窗口函数支持晚于 PostgreSQL

## 结论
JSON 存储需求是核心驱动因素，PostgreSQL 的 JSONB 索引性能显著优于
MySQL 的 JSON 列。学习成本可接受。
```

### 服务间通信方式

```markdown
# ADR-003：服务间通信采用 REST 而非 gRPC

**状态**：已接受

## 背景
系统拆分为多个服务后，需要确定服务间通信协议。
当前团队规模 8 人，服务数量预计 5-8 个。

## 决策
服务间通信采用 REST + JSON，不引入 gRPC。

## 结论
在当前团队规模下，gRPC 的性能优势（约 30% 序列化效率提升）
不足以抵消其带来的工具链复杂度和调试成本。
当 QPS 超过 5000 或服务数量超过 15 个时，重新评估此决策。
```

## 注意事项

- ADR 一旦"已接受"，**不要修改其内容**，应新建"取代"状态的 ADR
- 记录"为什么放弃"比记录"为什么选择"更重要，避免未来重蹈覆辙
- 决策不需要完美，记录当时的约束条件即可
