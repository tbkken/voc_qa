# Code Review

系统化检查代码质量，给出可执行的改进建议。

## Review 维度 & 优先级

| 优先级 | 维度 | 说明 |
|--------|------|------|
| P0 🔴 | 安全 | 注入、越权、敏感信息泄露 |
| P0 🔴 | 正确性 | 逻辑错误、边界条件遗漏、数据丢失风险 |
| P1 🟡 | 性能 | N+1 查询、不必要的循环、内存泄漏 |
| P1 🟡 | 可靠性 | 异常处理、事务边界、并发安全 |
| P2 🟢 | 可读性 | 命名、注释、函数长度 |
| P2 🟢 | 规范 | 是否符合 lang-style 规范 |

---

## 各语言 Review 检查项

### Python

**P0 安全**
- [ ] SQL 拼接 → 必须用参数化查询或 ORM
- [ ] `eval()` / `exec()` 使用 → 基本禁用
- [ ] 敏感信息（密码、token）是否出现在日志或响应里

**P0 正确性**
- [ ] None 判断：可为 None 的变量是否有防护
- [ ] 列表/字典修改：是否在迭代中修改集合
- [ ] 文件/连接：是否有 `with` 语句确保关闭

**P1 性能**
- [ ] 数据库查询：是否在循环里查询（N+1 问题）
- [ ] 大数据集：是否用生成器替代一次性加载列表
- [ ] 缓存：频繁读取的静态数据是否有缓存

**P1 可靠性**
- [ ] 异常捕获：是否捕获具体异常（禁止裸 `except:`）
- [ ] 事务：涉及多表写入是否在同一事务内

---

### Java / Kotlin

**P0 安全**
- [ ] SQL 注入：禁止字符串拼接 SQL，使用 JPA/MyBatis 参数绑定
- [ ] 越权访问：接口是否校验当前用户有权访问目标资源
- [ ] 反序列化：`ObjectInputStream` 使用场景是否安全

**P0 正确性**
- [ ] NPE 防护（Java）：返回值是否用 `Optional` 包装
- [ ] Kotlin 可空性：`!!` 使用是否有充分理由
- [ ] 事务注解：`@Transactional` 是否在 `public` 方法上，自调用是否失效

**P1 性能**
- [ ] JPA N+1：关联查询是否用 `fetch join` 或 `@EntityGraph`
- [ ] 分页：列表接口是否强制分页，禁止 `findAll()` 无限制查询
- [ ] 连接池：数据库/Redis 连接是否通过连接池管理

**P1 可靠性**
- [ ] 事务边界：Service 层事务是否正确，不在 Controller 开事务
- [ ] 并发安全：共享状态是否有同步机制
- [ ] 幂等性：POST 接口是否有防重复提交机制

---

### TypeScript / JavaScript

**P0 安全**
- [ ] XSS：用户输入是否直接渲染到 DOM（`dangerouslySetInnerHTML`）
- [ ] 敏感信息：token/密码是否存在 `localStorage`（应用 `httpOnly cookie`）
- [ ] 依赖安全：`npm audit` 是否有高危漏洞

**P0 正确性**
- [ ] 异步错误处理：`async/await` 是否有 `try/catch`
- [ ] 类型断言：`as` 的使用是否有运行时保证
- [ ] 副作用：`useEffect` 依赖数组是否完整

**P1 性能**
- [ ] 重渲染：组件是否因为不必要的引用变化触发重渲染
- [ ] 懒加载：大型组件/页面是否用 `React.lazy` 按需加载
- [ ] 列表渲染：是否有稳定的 `key`

## 输出格式

```markdown
## Code Review 结果

**总体评价**：[一句话概括，如"整体结构清晰，有 2 个需要修复的问题"]

### 🔴 必须修复（P0）

**[文件名:行号]** [问题标题]
```[语言]
// 问题代码
[具体代码]
```
**问题**：[解释为什么有问题]
**建议**：
```[语言]
// 修复后
[具体代码]
```

---

### 🟡 建议改进（P1）

**[文件名:行号]** [问题标题]
[简要说明 + 改进建议]

---

### 🟢 可选优化（P2）

- [文件名:行号]：[简短建议]

---

### ✅ 做得好的地方

- [具体的好的实践，鼓励继续保持]
```

## 注意事项

- 每个问题必须给出**具体**的改进代码，不能只说"这里有问题"
- P0 问题必须修复后才能合并，P1/P2 可协商
- Review 目的是提升代码质量，不是批评作者，措辞保持建设性
