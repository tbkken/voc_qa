# Postmortem

将故障转化为系统改进的机会。无指责，聚焦流程和系统改进。

## 核心原则

> 参与故障处理的人，在当时信息和工具条件下，做出了他们能做的最好决策。
> 复盘的目的是改进系统，不是追责个人。

---

文件路径：`docs/postmortem/INC-YYYY-NNN.md`

## 一句话摘要

v1.3.0 引入了缺少索引的慢查询，导致数据库连接池耗尽，支付接口不可用 22 分钟，
通过回滚 v1.2.3 恢复。

## 根因分析（5 Whys）

**表象**：支付接口大量超时报错

→ **Why 1**：数据库连接池被耗尽
→ **Why 2**：有大量慢查询持有连接不释放
→ **Why 3**：v1.3.0 新增了一个全表扫描的查询（缺少索引）
→ **Why 4**：Code Review 时没有发现这个查询性能问题
→ **Why 5（根因）**：没有在 CI 中对新增 SQL 做 EXPLAIN 检查，Code Review 清单也没有覆盖 SQL 性能

## 做得好的地方

- 监控在故障发生 2 分钟内触发告警，响应及时

- 故障期间按时同步进展，相关方及时知情

## 经验教训

**1. Staging 数据量不足会掩盖查询性能问题**
Staging 环境只有几百条测试数据，全表扫描感知不到。需要定期用脱敏生产数据测试。

**2. 连接池耗尽是典型雪崩**
慢查询 → 连接被占满 → 新请求排队等连接 → 超时 → 更多错误堆积。
早期告警（连接使用率 > 80%）能提前发现。

## 写复盘的时机和要求

| 级别 | 是否必须 | 截止时间 |
|------|----------|----------|
| P0 | 必须 | 故障恢复后 24 小时内完成初稿 |
| P1 | 建议 | 3 天内完成 |
| P2 | 可选 | 有值得沉淀的经验则写 |

**写复盘的好时机是故障刚恢复后**——记忆最清晰，时间线最准确，趁热打铁。

## 改进措施跟进

复盘写完不是终点。每次 Sprint 回顾时检查改进措施的完成情况，确保措施真的落地，而不是写完就忘。
