# Incident Review

将事故讨论转化为结构化复盘报告，防止同类问题再次发生。

```markdown

**状态**：已解决
**撰写人**：@xxx
**复盘日期**：YYYY-MM-DD

## 事故摘要

[2-3 句话描述：什么服务、出了什么问题、影响了什么、持续多久]

导致用户无法完成下单，影响约 2300 名用户，造成直接损失约 xx 元。

## 事故时间线

| 时间 | 事件 |
|------|------|
| 10:23 | 监控告警：订单服务错误率超过 5% |
| 10:28 | @xxx 开始排查，查看日志发现大量 `Connection pool exhausted` |
| 10:35 | 确认是数据库连接池耗尽，连接数从正常 20 飙升至 100 |
| 10:42 | 发现 v1.3.0 发布的新功能在某些场景下未关闭数据库连接 |
| 10:55 | 临时措施：重启服务，连接池恢复正常，错误率下降 |
| 11:10 | 发布 hotfix v1.3.1，修复连接泄漏 |
| 11:47 | 监控确认错误率恢复正常，事故结束 |

---

## 根本原因分析（5 Why）

**表面现象**：订单服务大量超时

**Why 1**：为什么超时？
→ 数据库连接池耗尽（max 100 个连接全部占用）

**Why 2**：为什么连接池耗尽？
→ 连接没有被正确释放，导致连接泄漏

**Why 3**：为什么连接没有释放？
→ v1.3.0 新增的导出功能，在异常分支中没有关闭数据库 Session

**Why 4**：为什么代码中有未关闭的 Session？
→ Code Review 没有发现这个问题；导出功能的异常路径没有写测试

**Why 5**：为什么没有写测试/没有被 Review 发现？
→ 时间压力下跳过了对异常路径的测试；Reviewer 只关注了主流程

**根本原因**：研发流程中缺少对异常路径的测试要求，且 Code Review 未包含资源释放检查项。

## 处理过程

### 临时措施（止血）
- 10:55：重启服务，释放积累的泄漏连接

### 根本修复
- 11:10：发布 hotfix，使用 `try-with-resources` / `use {}` 确保 Session 关闭
  ```kotlin
  // 修复后
  fun exportData(): ByteArray {
      return dataSource.connection.use { conn ->  // use{} 确保关闭
          // 导出逻辑
      }
  }
  ```

## 经验教训

**做得好的地方**：
- 告警及时，10:23 故障发生 5 分钟内收到告警
- 临时止血决策快，10:55 重启服务

**需要改进的地方**：
- 排查时间较长（从告警到定位根因花了 19 分钟），需要更好的日志可观测性
- 没有提前发现：发布流程中缺少连接泄漏检测

**这次学到了什么**：
- 任何涉及资源（数据库连接、文件句柄、HTTP 连接）的代码，必须用 try-with-resources 模式
- 连接池监控要前置告警，不能等耗尽了才知道

## 事故等级定义

| 等级 | 定义 | 响应时间 |
|------|------|----------|
| P0 | 核心功能（登录/下单/支付）完全不可用 | 立即响应，15 分钟内有进展 |
| P1 | 核心功能部分受损，或非核心功能不可用 | 30 分钟内响应 |
| P2 | 性能下降但功能可用，或影响极小范围用户 | 工作时间内处理 |

## 复盘注意事项

- **无指责原则**：复盘目的是改进系统和流程，不是追责个人
- **关注系统性问题**：单个人的失误背后一定有系统/流程的漏洞
- **改进措施要可落地**：每一条有明确的负责人和截止日期，不说"以后要注意"
- **复盘会议在 48 小时内举行**：趁记忆清晰，细节不遗漏
