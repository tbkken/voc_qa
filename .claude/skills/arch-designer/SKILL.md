# Arch Designer

输出可落地的系统架构设计。

## 第一步：收集约束条件

开始前先明确（未知标 TBD）：
- **规模预期**：DAU / 峰值 QPS / 数据量级
- **可用性**：99.9%（月停机 < 43 分钟）还是 99.99%
- **团队规模**：影响服务拆分粒度，5 人团队不拆微服务
- **现有技术栈**：已有哪些基础设施
- **时间约束**：MVP 快速验证还是长期演进

## 第二步：输出架构文档模板

```markdown

## 1. 系统概述
[一段话：系统是什么、解决什么问题、核心用户]

**设计目标**：
| 目标 | 指标 |
|------|------|
| 可用性 | 99.9% |
| 响应时间 | P99 < 500ms |
| 峰值吞吐 | xxx QPS |

## 2. 逻辑架构

### 分层与模块

| 模块 | 职责 | 技术选型 | 选型理由 |
|------|------|----------|----------|
| 接入层 | 路由、限流、鉴权 | Nginx | 团队熟悉，运维成本低 |
| 用户服务 | 注册/登录/权限 | Spring Boot (Kotlin) | 主力后端栈 |
| 业务服务 | 核心业务逻辑 | Spring Boot / FastAPI | 视复杂度 |
| 主数据库 | 结构化数据 | PostgreSQL | ACID + JSON 支持 |
| 缓存 | 热点数据/Session | Redis | 标准选型 |
| 文件存储 | 图片/附件 | OSS/MinIO | 成本低，CDN 友好 |

### 服务交互图
[描述核心服务间调用关系和数据流向]

## 3. 数据架构

**选型原则**：
- 关系型数据 → PostgreSQL
- 缓存/排行榜/Session → Redis
- 非结构化文档 → PostgreSQL JSONB（量小）或 MongoDB（量大）
- 文件/图片 → 对象存储

**一致性策略**：
| 场景 | 策略 |
|------|------|
| 单服务多表写 | 数据库事务 |
| 跨服务数据同步 | 最终一致性 + 消息队列 |
| 缓存与数据库 | Cache-Aside，TTL + 主动失效 |

## 4. 部署架构（Docker + 云服务器）

```
[负载均衡 Nginx]
      │
 ┌────┴────┐
[App-1] [App-2]     ← 最少 2 实例保高可用
      │
[PostgreSQL 主从]   ← 主写从读
[Redis Sentinel]    ← 主从 + 哨兵
```

**环境规划**：
| 环境 | 用途 | 规格 |
|------|------|------|
| local | 本地开发 | docker-compose 单实例 |
| staging | 集成/验收 | 单实例，同生产配置 |
| production | 线上 | 多实例，主从DB |

## 5. 关键技术决策

| 编号 | 决策点 | 选择 | 放弃 | 理由 |
|------|--------|------|------|------|
| D-01 | 后端框架 | Spring Boot | Quarkus | 团队熟悉度 |
| D-02 | 数据库 | PostgreSQL | MySQL | JSON 支持更好 |
| D-03 | 服务通信 | REST | gRPC | 简单，当前规模够用 |

## 6. 风险与演进路径

**当前架构局限**：
- QPS > xxx 时需拆分 [具体服务]
- 写入 > xxx/s 时考虑读写分离

**演进路径**：
MVP 单体 → 按业务域拆服务 → 引入消息队列解耦 → 按需水平扩展
```

## 设计原则

1. **够用即可**：按当前团队规模和量级选最简方案，5 人团队不上 K8s
2. **单一职责**：每个服务只做一件事
3. **失败设计**：每个外部调用都要考虑超时、重试、降级
4. **可观测性**：架构中预留日志、指标、链路追踪位置
5. **渐进演进**：明确当前方案的边界，超过边界时的演进路径
