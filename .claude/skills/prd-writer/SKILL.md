# PRD Writer

将需求描述转化为供业务方确认的 PRD，自然语言，聚焦"为什么做"和"做什么"。

## 定位

PRD 是给**人**看的，不是给 AI 执行的：
- 读者：产品经理、业务方、设计师、管理层
- 语言：自然语言，不需要 Given/When/Then
- 目标：对齐需求范围，达成共识
- 下一步：PRD 确认后，由 `spec-writer` 转化为 Spec

---

## 执行步骤

### 第一步：需求澄清

收到需求描述后，确认以下信息（无法确认的直接标 TBD，不要停下来等待）：

1. **用户是谁**：哪类用户使用这个功能？
2. **核心场景**：用户最典型的使用路径是什么？
3. **边界**：本期明确不做哪些？
4. **成功标准**：怎么判断这个功能做好了？
5. **约束**：有无性能、安全、兼容性要求？

### 第二步：生成 PRD

```markdown
# PRD: [功能名称]

**版本**：v1.0
**状态**：草稿 / 评审中 / 已确认
**创建日期**：YYYY-MM-DD
**产品负责人**：@xxx

---

## 背景 & 目标

[一段话说清楚为什么要做，解决什么痛点，当前状况是什么]

**目标**：
- [可量化的目标，如"将下单成功率从 85% 提升到 95%"]
- [目标2]

**非目标**（本期明确不做）：
- [具体排除的内容，防止范围蔓延]

---

## 用户故事

### [故事标题1]

**作为** [用户角色]，
**我希望** [能做什么]，
**以便** [达到什么目的]。

**主要场景**：
[用自然语言描述用户操作路径，1-3 个步骤]

**异常情况**：
[可能遇到的错误或特殊情况，自然语言描述即可，不需要穷举]

---

### [故事标题2]

**作为** [用户角色]，
**我希望** [能做什么]，
**以便** [达到什么目的]。

**主要场景**：
[描述]

---

## 功能说明

### [子功能1]

[自然语言描述功能行为，说清楚用户能做什么、系统会怎么响应]

### [子功能2]

[描述]

---

## 非功能需求

| 类型 | 要求 |
|------|------|
| 性能 | [如"列表页 2 秒内加载完成"] |
| 安全 | [如"需要登录，数据按用户隔离"] |
| 兼容性 | [如"支持 Chrome 90+、iOS 14+"] |

---

## 依赖 & 风险

**外部依赖**：
- [依赖的外部服务、团队、接口，预计何时可用]

**风险**：
- [潜在风险] → 应对方案：[...]

---

## 开放问题（待确认）

| # | 问题 | 负责人 | 截止日期 |
|---|------|--------|----------|
| Q1 | [待确认的问题] | @xxx | xx/xx |
```

### 第三步：输出后的提示

PRD 生成后，告知用户：

```
PRD 已生成。建议步骤：
1. 与产品/业务方确认内容，重点确认：
   - 用户故事是否完整覆盖核心场景
   - 非目标是否明确排除了容易蔓延的内容
   - 开放问题是否有负责人跟进
2. PRD 确认后，对我说"把这份 PRD 转成 Spec"
   我会将 PRD 交给 spec-writer，自动转化为带
   Given/When/Then 验收标准的 Spec，供开发和 Claude Code 执行。
```
