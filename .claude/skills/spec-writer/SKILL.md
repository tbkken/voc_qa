# Spec Writer

将 PRD 转化为 SDD 标准的 Spec，让 Claude Code 和开发可以直接执行。

## 定位

Spec 是给 **Claude Code 和开发**执行的，不是给业务方看的：
- 读者：Claude Code、开发工程师、测试工程师
- 语言：结构化，Given/When/Then，有唯一编号
- 目标：消除歧义，让每条验收标准都可以直接转成测试用例
- 前置：必须有已确认的 PRD（由 `prd-writer` 生成或用户提供）

---

## PRD → Spec 转化规则

| PRD 内容 | 转化为 Spec |
|----------|------------|
| 用户故事（自然语言） | 用户故事 + Given/When/Then AC，加 US-xx 编号 |
| 主要场景描述 | 正常路径 AC（标注"正常"） |
| 异常情况描述 | 异常路径 AC（标注"异常"） |
| 隐含但未写出的边界 | **主动补全**边界 AC（标注"边界"） |
| 功能说明 | 数据需求 + 接口需求 |
| 非功能需求 | 直接保留，补充关联 AC 编号 |

**最重要的一步：主动补全 PRD 未覆盖的场景。**

PRD 通常只描述 happy path，转化时必须系统性补出：

| PRD 写了 | Spec 必须补出 |
|----------|--------------|
| 用户可以登录 | 密码错误的提示、连续失败锁定、账号不存在 |
| 展示订单列表 | 空列表状态、加载失败、网络超时 |
| 用户可以提交表单 | 必填项为空、格式校验失败、重复提交 |
| 支持搜索 | 无搜索结果、搜索词过长、特殊字符 |

## 执行步骤

### 第一步：解析 PRD

从 PRD 中提取：
- 所有用户故事（标注为 US-01、US-02...）
- 每个故事的主要场景和异常情况
- 非功能需求
- 开放问题（未确认的项目标 TBD）

### 第二步：生成 Spec 文档

```markdown
# Spec: [功能名称]

**Spec 编号**：SPEC-[序号]
**关联 PRD**：[PRD 标题或链接]
**版本**：v1.0
**状态**：草稿 / 评审中 / 已确认
**创建日期**：YYYY-MM-DD

---

## 背景 & 目标

[直接引用 PRD 的背景，不重复写，一句话说明来源]

**非目标**（本期明确不做）：
- [从 PRD 继承]

---

AC 编写规范：
- 编号全文档唯一且连续：AC-01、AC-02...

- When：用户执行的具体操作
```

### 第三步：输出后的检查与提示

Spec 生成后，逐项自检：

- [ ] 每个用户故事至少有 1 个正常 AC、1 个异常 AC
- [ ] PRD 中隐含的边界情况已全部补出
- [ ] 所有 AC 的 Then 描述的是可观测行为（不是"系统正常响应"）
- [ ] AC 汇总表完整，编号连续无跳号

然后输出：

```
Spec 已生成，共 [N] 个用户故事，[M] 条验收标准。

补全了 PRD 中未明确的场景：
- [列出主动补全的 AC，让用户知道补了什么]

TBD 项目（需确认后才能进入开发）：
- [列出标注 TBD 的内容]

Spec 确认后，对我说"根据 SPEC-[编号] 生成 Tasks"
ticket-mgr 会自动读取 AC 编号，生成与 Spec 完全追溯的任务列表。
```

## Given/When/Then 质量标准

**不合格**（Then 不可观测）：
```
- [ ] AC-01：Given 用户已登录，When 提交表单，Then 系统处理成功
```

**合格**（Then 具体可测）：
```
- [ ] AC-01（正常）：Given 用户已登录且所有必填项已填写，When 点击"提交订单"按钮，
  Then 页面跳转到订单详情页，URL 变为 /orders/{新订单ID}，页面显示"订单提交成功"，
  订单状态为"待支付"
```

Then 必须能回答：**测试工程师拿到这条 AC，不需要问任何人就能写出测试用例。**
