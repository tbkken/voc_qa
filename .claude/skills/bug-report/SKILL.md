# Bug Report

两阶段设计：先分析给人看，人决策后再执行。

---

## 阶段一：分析（自动执行）

收到 bug 描述后，自动完成以下分析，**输出报告后暂停，等待人工决策**。

### 1. 分配编号

读取 `docs/bugs/INDEX.md`，取当前最大编号 +1。
若 INDEX 不存在，从 BUG-001 开始。

### 2. 判断严重程度

按以下规则自动评估，输出建议级别（人可修改）：

| 级别 | 判断条件 |
|------|----------|
| P0 | 核心主流程完全不可用；数据丢失或错误写入；安全漏洞 |
| P1 | 核心功能部分异常；影响多数用户；数据计算错误 |
| P2 | 非核心功能异常；边界情况缺失；UI 显示错误但不影响操作 |
| P3 | 体验问题；文案错误；样式偏差；极低频场景 |

### 3. 定位根因

从报错信息、代码、日志定位：问题出在哪一层，具体是哪个文件哪个函数，为什么出了这个问题。

### 4. 查 Spec，判断分流

打开对应 Spec 的 AC 汇总表，查找这个 bug 的场景：

- **情况 A：代码写错了** — AC 存在，代码实现与 AC 不符
- **情况 B：Spec 遗漏场景** — 这个场景 AC 汇总表里没有
- **情况 C：需求有歧义** — AC 本身描述不清楚或业务逻辑矛盾
- **情况 D：无对应 Spec** — 功能没有 Spec 文档（老代码等情况）

### 5. 输出分析报告

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Bug 分析报告 · BUG-[编号]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📋 基本信息
  编号：BUG-[编号]
  标题：[一句话概括]
  日期：YYYY-MM-DD
  严重程度：P[x]（[判断理由，一句话]）

🔍 现象
  描述：[用户描述的现象]
  触发路径：[操作步骤]
  期望行为：[正确应该是什么]
  实际行为：[当前是什么]

⚡ 根因定位
  位置：[文件名:函数名:行号]
  原因：[为什么出了这个问题]

📑 Spec 分析
  关联 Spec：[SPEC-编号，或"无 Spec"]
  分流结果：[情况 A/B/C/D]
  说明：[具体说明，如"AC-05 有定义但代码未正确实现"]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请选择要执行的动作（多选，空格分隔）：

  [A] 补充 / 更新 Spec AC
      在 Spec 里新增或修正对应的验收标准
      （情况 A 时：标注 AC 已修复；情况 B 时：新增 AC）

  [B] 修复代码
      按根因定位修复对应文件

  [C] 补充测试
      根据 bug 性质自动选择测试层次（单元/集成/接口）

  [D] 生成 bug 记录
      输出 Markdown + JSON 文件，更新 INDEX

示例输入：
  A B C D  →  全部执行（完整修复）
  B D      →  修代码 + 生成记录，不动 Spec，不补测试
  A B      →  补 AC + 修代码，暂不补测试
  D        →  只记录，什么都不改（挂起或忽略）
  直接回车  →  取消，什么都不做

请输入字母（A B C D 任意组合，或直接回车取消）：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 阶段二：执行（按选择的字母组合）

收到字母组合后，**只执行用户选择的动作**，未选择的动作完全跳过。

---

### [A] 补充 / 更新 Spec AC

根据分流情况执行：

**情况 A（代码写错）**：
```
- 不新增 AC
- 修复后在对应 AC 旁标注：
  [已修复] YYYY-MM-DD BUG-[编号]
```

**情况 B（Spec 遗漏）**：
```
- 在对应 US 下新增 AC，编号续接最大编号：
  - [ ] AC-[编号]（异常/边界）：Given...，When...，Then...
- 更新 AC 汇总表
- 生成修复 Task（ticket-mgr 逻辑），关联新 AC
- 修复后标注：[已修复] YYYY-MM-DD BUG-[编号]，由自测新增
```

**情况 C（需求歧义）**：
```
- 暂停，输出歧义分析，等待人工确认：

  ⚠️ 需求歧义，请确认后继续

  PRD 原文：[相关描述]
  选项 X：[理解X] → 对应实现：[...]
  选项 Y：[理解Y] → 对应实现：[...]

  请回复 X 或 Y，确认后继续执行其他已选动作。
```

**情况 D（无 Spec）**：
```
- 提示：该功能无对应 Spec，建议先用 spec-writer 补充
- 询问是否继续执行其他已选动作
```

---

### [B] 修复代码

按根因定位，修复对应文件。

---

### [C] 补充测试

根据 bug 性质自动判断测试层次：

| bug 性质 | 测试类型 |
|----------|---------|
| 纯逻辑错误 | 单元测试（unit-test-gen） |
| 涉及 DB/Redis/MQ | 集成测试（integration-test-gen） |
| 接口层问题 | 接口测试（api-test-gen） |
| 前端组件 | 前端单元测试或 e2e |

测试命名：`test_[函数名]_[bug场景]_[期望行为]`

---

### [D] 生成 bug 记录

生成 Markdown + JSON 文件，更新 INDEX。

**JSON 中记录实际执行的动作**：
```json
"resolution": {
  "spec_updated": true,   // 是否执行了 A
  "code_fixed": true,     // 是否执行了 B
  "test_added": false,    // 是否执行了 C
  "note": "用户选择跳过测试补充"
}
```

**status 根据实际情况自动判断**：
- 选了 B 且修复完成 → `"fixed"`
- 没选 B → `"ignored"`（如果是主动忽略）或 `"pending"`（如果是暂不处理）
- 询问用户：这个 bug 是"忽略"还是"待后续处理"？

---

### 直接回车（取消）

不生成任何文件，直接结束。

---

## 记录文件格式

### 目录结构

```
docs/bugs/
├── INDEX.md                                    ← 汇总索引
├── BUG-001-2024-03-15-订单金额计算错误.md      ← Markdown 记录
├── BUG-001-2024-03-15-订单金额计算错误.json    ← JSON 记录
├── BUG-002-2024-03-15-重复提交产生重复订单.md
├── BUG-002-2024-03-15-重复提交产生重复订单.json
└── ...
```

### Markdown 文件

```markdown
# BUG-[编号]：[标题]

**日期**：YYYY-MM-DD
**严重程度**：P[x]
**状态**：已修复 / 已忽略 / 待处理
**处理方式**：完整修复 / 仅修代码 / 记录忽略 / 暂不处理

---

## 现象

**描述**：[用户原始描述]
**触发路径**：[操作步骤]
**期望行为**：[正确应该是什么]
**实际行为**：[当前是什么]

## 根因分析

**位置**：[文件:函数:行号]
**原因**：[为什么出了这个问题]

## Spec 变更

**关联 Spec**：[SPEC-编号 / 无]
**变更类型**：无变更 / 新增 AC / 更新 AC / 跳过
**变更内容**：[具体内容，或"无"，或"用户选择跳过"]

## 代码变更

[文件路径]：[改了什么]

## 新增测试

[测试文件路径]：[测试用例名称]
（或：用户选择跳过）

## 验证方式

[如何确认 bug 已修复]

## 备注

[忽略原因 / 其他说明]
```

### JSON 文件（用于导入内部系统）

```json
{
  "id": "BUG-001",
  "title": "订单金额计算错误",
  "date": "2024-03-15",
  "severity": "P1",
  "status": "fixed",
  "resolution": "full",

  "symptom": {
    "description": "两件商品各 99 元，总价显示 199 而不是 198",
    "trigger_path": "创建订单 → 选择两件商品 → 查看合计",
    "expected": "总价应为 198.00",
    "actual": "总价显示 199.00"
  },

  "root_cause": {
    "location": "src/service/OrderService.kt:calculateTotal:42",
    "analysis": "BigDecimal 加法中使用了 double 中间类型导致精度丢失",
    "category": "code_error"
  },

  "spec_change": {
    "type": "none",
    "spec_ref": "SPEC-002",
    "ac_ref": "AC-05",
    "details": "AC-05 定义正确，代码实现有误，已标注修复",
    "new_acs": []
  },

  "code_change": {
    "files": ["src/service/OrderService.kt"],
    "summary": "修改 calculateTotal，改用 BigDecimal.add() 避免精度丢失"
  },

  "tests_added": [
    {
      "type": "unit",
      "file": "src/test/OrderServiceTest.kt",
      "case": "test_calculateTotal_multipleItems_returnsExactSum"
    }
  ],

  "verification": "运行 OrderServiceTest 全部通过",
  "notes": ""
}
```

**`status` 枚举值**：`fixed` / `ignored` / `pending`

**`resolution` 枚举值**：`full`（完整修复）/ `code_only`（仅修代码）/ `ignored`（忽略）/ `pending`（暂不处理）/ `cancelled`（取消）

**`spec_change.type` 枚举值**：`none`（无变更）/ `ac_added`（新增 AC）/ `ac_updated`（更新 AC）/ `prd_updated`（更新 PRD）/ `skipped`（用户跳过）

## 使用示例

```
你：提个 bug，订单金额不对，两件 99 的商品，
    总价显示 199

→ 阶段一自动执行：
  - 编号：BUG-001
  - 严重程度：P1（数据计算错误）
  - 根因：OrderService.calculateTotal，BigDecimal 精度问题
  - Spec：SPEC-002，情况 A，AC-05 有定义但代码实现有误
  - 输出分析报告，列出 1-5 选项，暂停等待

你：1（完整修复）

→ 阶段二执行：
  - 修复 OrderService.kt
  - AC-05 标注已修复
  - 生成单元测试
  - 生成 BUG-001.md 和 BUG-001.json
  - 更新 INDEX.md
  - 输出完成摘要
```

```
你：提个 bug，这个页面标题少了个字，不影响使用

→ 阶段一：P3，前端文案问题，无 Spec 对应
→ 你：3（记录忽略）
→ 阶段二：只生成记录文件，忽略原因写"不影响使用，文案问题"
           更新 INDEX，状态标记已忽略
```
