# Ticket Manager

把 Spec 或需求拆成可独立执行的原子任务，每个 Task 都能追溯到具体的 AC。

## 两种使用场景

### 场景 A：有 Spec 文档（SDD 标准流程）

用户提供了 `spec-writer` 生成的 Spec 文档，按以下步骤执行：

1. 读取 Spec 中的"AC 汇总表"，获取所有 AC 编号
2. 按用户故事分组，将 AC 映射到具体的实现任务
3. 每个 Task 必须标注覆盖的 AC 编号
4. 输出结束时，输出"AC 覆盖矩阵"，确认所有 AC 都有对应 Task

### 场景 B：无 Spec 文档（通用拆解）

用户直接描述需求，按传统方式拆解，输出结果提示建议先用 `spec-writer` 生成 Spec 再拆 Tasks。

---

## SDD 场景下的 Task 拆解步骤

### 第一步：解析 Spec

从 Spec 文档提取：
- 所有用户故事编号（US-xx）和标题
- 所有 AC 编号和简述
- 接口需求和数据需求

### 第二步：按层次生成 Tasks

每个用户故事展开为多个 Task，按技术层次分：

```
US-01：[用户故事标题]
│
├── Task 1.1 [数据层]  数据库表设计 + 迁移脚本
│   覆盖 AC：（数据结构类 AC）
│
├── Task 1.2 [后端]   业务接口实现
│   覆盖 AC：AC-01、AC-02
│
├── Task 1.3 [后端]   异常处理与边界情况
│   覆盖 AC：AC-03、AC-04
│
├── Task 1.4 [前端]   页面实现
│   覆盖 AC：AC-01（前端部分）
│
└── Task 1.5 [测试]   单元测试 + 联调
    覆盖 AC：AC-01 ~ AC-04（全部验证）
```

### 第三步：输出 AC 覆盖矩阵

```
AC 覆盖矩阵：

| AC 编号 | 简述 | 覆盖 Task | 状态 |
|---------|------|-----------|------|
| AC-01 | 正常登录流程 | Task 1.2、Task 1.5 | ✅ 已覆盖 |
| AC-02 | 登录态保持 | Task 1.2、Task 1.5 | ✅ 已覆盖 |
| AC-03 | 密码错误锁定 | Task 1.3、Task 1.5 | ✅ 已覆盖 |
| AC-04 | 登录态续期 | Task 1.3 | ✅ 已覆盖 |

未覆盖 AC：无
```

如发现有 AC 未被任何 Task 覆盖，**必须补充 Task**，不允许遗漏。

## Task 描述模板（SDD 版）

```markdown
## [PROJ-xxx] [Task 标题]

**类型**：Feature / Bug / Refactor / Chore
**优先级**：P0 / P1 / P2
**估时**：X 天
**负责人**：@xxx
**所属 Spec**：SPEC-[编号]
**所属故事**：US-[编号] [故事标题]

### 任务描述
[具体做什么，越具体越好]

### 覆盖的验收标准
<!-- 直接从 Spec 复制对应 AC，不要重新描述 -->
- [ ] **AC-01**（正常）：Given ...，When ...，Then ...
- [ ] **AC-03**（异常）：Given ...，When ...，Then ...

### 技术说明
[实现思路、影响到的模块、需要注意的技术点]

### 依赖
- 依赖 Task xxx 完成后才能开始
- 需要 @xxx 提供 xxx

### 完成标准
- [ ] 所有覆盖的 AC 均通过验证
- [ ] 单元测试覆盖率 ≥ 70%
- [ ] Jenkins CI 通过
- [ ] Code Review 通过
```

## 好任务的标准（INVEST）

| 标准 | 含义 |
|------|------|
| Independent | 可独立开发，不强依赖未完成的其他任务 |
| Negotiable | 范围可以协商 |
| Valuable | 完成后有独立的可验证价值（对应至少一条 AC） |
| Estimable | 能估出工时 |
| Small | 1-3 天能完成，超过 3 天继续拆 |
| Testable | 有明确完成标准（来自 Spec 的 AC） |

## 工时估算参考

| 任务类型 | 参考工时 |
|----------|----------|
| 数据库表设计 + Flyway 迁移脚本 | 0.5d |
| 简单 CRUD 接口（含单元测试） | 0.5–1d |
| 含复杂业务逻辑的接口（状态机/事务/幂等） | 1.5–2d |
| 前端列表页（含搜索/分页/排序） | 1–1.5d |
| 前端表单页（含校验/提交/反馈） | 1–1.5d |
| 单元测试补充 | 0.5d/模块 |
| E2E 测试（一个完整用户流程） | 0.5–1d |
| 联调 + Bug 修复缓冲 | 0.5d/接口 |

估算原则：估乐观时间 ×1.5 作为实际工时，超过 2d 的任务继续拆。
