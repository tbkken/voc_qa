# Spec: 洞察分析功能

**Spec 编号**：SPEC-001
**关联 PRD**：洞察分析功能需求（用户口述）
**版本**：v1.0
**状态**：已确认
**创建日期**：2026-04-22

---

## 背景 & 目标

在现有 VoC 问答系统（自然语言 → SQL → DuckDB → LLM 叙述 → 图表）的基础上，新增"洞察分析"模式：用户一次性输入多个数据需求，系统串行处理、逐步渲染，最终输出由所有数据支撑的核心洞察总结。

**非目标**（本期明确不做）：
- 并发/并行处理多个数据需求
- 洞察结果的持久化存储或导出
- 用户自定义分析维度模板

---

## 用户故事与验收标准

### US-01：识别洞察分析意图并解析需求列表

**作为** VoC 分析师，
**我希望** 输入一句包含多个数据需求的自然语言，系统自动识别为洞察分析模式并拆解需求，
**以便** 不需要逐条手动提问。

**验收标准**：

- [ ] **AC-01**（正常）：Given 用户输入任意问题，When 用户提交消息，Then 前端统一路由到 `/api/insight`，后端通过 **LLM 解析**提取背景条件字符串 + 数据需求列表（LLM 理解自然语言，按独立可查询需求拆分 items，如问号/句号/顿号分隔的多问题均应拆分为多条 items）；LLM 调用失败时降级为 `items=[question]` 单条，不中断流程。[已修复] 2026-04-23 BUG-001，统一走洞察 pipeline；[已修复] 2026-04-23 BUG-004，规则解析改为 LLM 解析

- [ ] **AC-02**（正常）：Given LLM 解析后需求列表有 N（N≥2）条，When 解析完成，Then 后端通过 SSE 首先推送一条 `type: init` 事件，payload 包含 `{ total: N, items: ["需求1", "需求2", ...] }`，前端据此在消息气泡顶部渲染占位区域（显示文案："🔍 正在洞察分析，还有 N 个数据待分析…"）。

- [ ] **AC-03**（正常）：Given 用户输入只包含 1 条数据需求，When 用户提交，Then 系统仍走洞察分析流程（单条视为特殊情况），解析结果 `total: 1`，正常执行后续流程。

- [ ] **AC-04**（异常）：Given 用户输入包含触发词但 LLM 解析后需求列表为空，When 解析完成，Then 后端推送 `type: error` 事件，文案："未能解析出有效的分析需求，请重新描述（如：分析以下细节：需求1、需求2）"，前端显示该错误气泡，流程终止。

- [ ] **AC-05**（异常）：Given LLM 解析阶段超时（超过 30s），When 超时触发，Then 后端推送 `type: error` 事件，文案："洞察解析超时，请稍后重试"，SSE 连接关闭，前端显示错误。

- [ ] **AC-06**（边界）：Given 用户输入不含列表结构（无"分析以下"等触发词），When 用户提交，Then 系统将整条问题作为单一数据需求（`items=[question]`）走洞察 pipeline，不存在独立的「普通问答」分支。[已修复] 2026-04-23 BUG-001，所有问题统一走洞察 pipeline

- [ ] **AC-07**（边界）：Given 需求列表解析出超过 10 条，When 解析完成，Then 系统正常处理所有条目，不截断（无上限限制）。

### US-02：串行逐步获取数据并实时渲染图表

**作为** VoC 分析师，
**我希望** 每处理完一个数据需求后立刻看到图表和分析，而不是等所有需求都完成才显示，
**以便** 在等待过程中可以先阅读已完成的分析。

**验收标准**：

- [ ] **AC-08**（正常）：Given 洞察分析已启动（收到 `init` 事件），When 后端开始处理第 i 个需求，Then 后端先推送 `type: item_start` 事件，payload `{ index: i, total: N, query: "需求i文本" }`，前端在消息气泡内追加一个「分析卡片」骨架（含 loading 状态，标题显示需求文本）。

- [ ] **AC-09**（正常）：Given 第 i 个需求的 SQL 执行 + LLM 叙述完成，When 后端组装好结果，Then 后端推送 `type: item_done` 事件，payload 包含 `{ index: i, sql, narration, chart_type, columns, rows }`；前端将对应卡片的 loading 替换为：叙述文本 + ECharts 图表（或表格），同时更新顶部占位区域文案："🔍 正在洞察分析，还有 (N-i) 个数据待分析…"。

- [ ] **AC-10**（正常）：Given 第 i 个需求处理完成（`item_done`），When 还有下一个需求，Then 后端**同步串行**处理第 i+1 个需求（不并发），每次只有一个 item 处于处理中状态。

- [ ] **AC-11**（异常）：Given 某个数据需求生成的 SQL 被 `sql_guard` 拒绝，When 拒绝发生，Then 后端推送 `type: item_error` 事件，payload `{ index: i, error: "SQL 安全校验未通过：{原因}", sql: "<LLM生成的原始SQL>" }`；前端对应卡片显示错误原因 + 实际 SQL（折叠代码块），方便用户判断生成了什么语句；**继续处理下一条需求**，不中断整体流程。[已修复] 2026-04-23 BUG-005，payload 补充 sql 字段

- [ ] **AC-12**（异常）：Given 某个数据需求执行 SQL 后返回空结果（0 行），When 叙述完成，Then LLM 叙述内容说明"未查询到符合条件的数据"，`item_done` 正常推送，chart_type 设为 `table`，rows 为空数组；前端对应卡片显示叙述文本 + 空表格提示"暂无数据"。

- [ ] **AC-13**（异常）：Given 某个数据需求的 LLM 叙述调用超时（超过 300s），When 超时，Then 后端推送 `type: item_error`，文案："第 i 项分析超时"，继续处理下一条。

- [ ] **AC-14**（边界）：Given 所有需求均处理完（无论成功或失败），When 最后一条处理结束，Then 顶部占位区域文案更新为："📝 所有数据已就绪，正在生成洞察总结…"，后端开始生成总结（进入 US-03 流程）。

- [ ] **AC-25**（正常）：Given SQL 执行返回数据，When LLM 生成叙述（narration），Then 叙述中引用的**所有数字、排名、占比必须直接来自查询结果表格**，禁止凭先验知识或训练数据编造；若查询结果为空（0 行），叙述只说明"未查询到符合条件的数据"，不给出任何推断性结论；禁止出现"通常"、"一般"、"可能"、"预计"等推测性表述。[已修复] 2026-04-23 BUG-006，由自测新增

### US-03：生成并流式输出核心洞察总结

**作为** VoC 分析师，
**我希望** 所有数据分析完成后，在顶部区域看到一份由所有数据支撑的核心洞察报告，
**以便** 快速获得本次多维分析的结论。

**验收标准**：

- [ ] **AC-15**（正常）：Given 所有 item 处理完成，When 后端调用 LLM 生成总结，Then 总结 prompt 包含所有成功的 `item_done` 数据（narration + 核心指标数字），LLM 输出格式为：每个洞察点独立成段，格式 `「观点标题」：{一句结论}，{数据佐证，含具体数字}`，共 3~5 个核心洞察点；**system prompt 必须以资深 VoC 分析专家为管理层撰写报告的角色定义，禁止输出引言、推理过程和收尾总结，每个结论必须有具体数据支撑，高管报告风格**。[已修复] 2026-04-23 BUG-003，由自测新增

- [ ] **AC-16**（正常）：Given LLM 生成总结内容，When 内容流式输出，Then 后端通过 SSE 推送 `type: summary_chunk` 事件（streaming），前端在顶部占位区域实时追加文本，呈现打字机效果；**后端在每个 chunk 推送前必须过滤推理模型（DeepSeek-R1、QwQ 等）输出的 `<think>…</think>` 块，确保前端不收到任何推理过程内容**。[已修复] 2026-04-23 BUG-002，由自测新增

- [ ] **AC-17**（正常）：Given 总结流式输出完毕，When 后端推送 `type: summary_done` 事件，Then 前端将顶部占位区域状态切换为"已完成"，移除 loading 动画，标题显示："📊 洞察总结"，SSE 连接关闭。

- [ ] **AC-18**（异常）：Given 所有 item 均失败（无成功数据），When 尝试生成总结，Then 后端推送 `type: summary_error`，文案："所有数据需求均分析失败，无法生成洞察总结"，顶部占位区域显示该错误，流程结束。

- [ ] **AC-19**（异常）：Given 总结生成时 LLM 调用失败，When 失败，Then 后端推送 `type: summary_error`，文案："洞察总结生成失败，但各项分析数据已在下方展示"，顶部占位区域显示该错误，各 item 卡片数据保持可见。

- [ ] **AC-20**（边界）：Given 部分 item 失败、部分成功，When 生成总结，Then 总结仅基于成功的 item 数据生成，失败 item 不纳入总结，但总结开头注明："以下洞察基于 M/N 项成功分析（{失败原因摘要}）"。

### US-04：前端消息气泡 UI 布局

**作为** VoC 分析师，
**我希望** 洞察分析的 UI 布局清晰、有层次感，
**以便** 区分总结区域和各项数据分析区域。

**验收标准**：

- [ ] **AC-21**（正常）：Given 洞察分析消息触发，When `init` 事件收到，Then 前端在对话流中渲染一个"洞察分析消息气泡"，气泡内布局从上到下为：① 顶部占位卡片（summary placeholder，含 loading spinner + 文案）；② 各 item 分析卡片区（按顺序排列，初始时全部为骨架/loading 状态）。注意：普通问答气泡不受影响。

- [ ] **AC-22**（正常）：Given item 分析卡片渲染，When `item_done` 收到，Then 对应卡片显示：① 卡片标题（需求文本，加序号，如"1. 各品类差评数量分布"）；② ECharts 图表（bar/line/metric 按 chart_type 渲染，宽度撑满卡片）；③ 叙述文本（灰色小字，图表下方）。

- [ ] **AC-23**（正常）：Given 洞察分析完成（`summary_done`），When 用户查看，Then 顶部总结卡片背景色有别于 item 卡片（如使用强调色边框或不同背景），标题"📊 洞察总结"加粗显示，内容支持 Markdown 渲染（加粗、换行）。

- [ ] **AC-24**（边界）：Given 洞察分析消息气泡内容较多，When 用户滚动，Then 页面滚动正常，不影响其他消息气泡的显示；新 item 卡片渲染时自动滚动到最新卡片（不滚动到最顶部）。

## 接口需求

- `POST /api/insight` (SSE) — 洞察分析入口，接收 `{ question: string }`，返回 SSE 事件流

SSE 事件类型汇总：

| 事件类型 | 触发时机 | payload 字段 |
|----------|----------|-------------|
| `init` | LLM 解析需求完成 | `{ total, background, items[] }` |
| `item_start` | 开始处理第 i 条 | `{ index, total, query }` |
| `item_done` | 第 i 条处理成功 | `{ index, sql, narration, chart_type, columns, rows }` |
| `item_error` | 第 i 条处理失败 | `{ index, error }` |
| `summary_chunk` | 总结流式输出 | `{ text }` |
| `summary_done` | 总结输出完成 | `{}` |
| `summary_error` | 总结生成失败 | `{ error }` |
| `error` | 全局错误（解析失败等） | `{ error }` |

---

## 数据需求

| 字段名 | 类型 | 说明 | 关联 AC |
|--------|------|------|---------|
| `question` | String | 用户原始输入 | AC-01 |
| `background` | String | LLM 解析出的背景条件（如"2026年1月份"） | AC-01 |
| `items` | String[] | 有序数据需求列表 | AC-02 |
| `index` | Int | 当前处理的需求序号（0-based） | AC-08 |
| `chart_type` | Enum | metric/bar/line/table | AC-09 |
| `narration` | String | LLM 叙述文本 | AC-09 |

---

## 非功能需求

| 类型 | 要求 | 关联 AC |
|------|------|---------|
| 串行性 | 同一洞察任务中需求必须串行处理，不得并发 | AC-10 |
| 容错性 | 单条需求失败不中断整体流程 | AC-11、AC-13 |
| 超时 | LLM 解析 ≤30s，单条需求叙述继承现有 LLM_TIMEOUT | AC-05、AC-13 |
| SSE 兼容 | 使用标准 `text/event-stream`，前端用 `EventSource` 或 `fetch` + ReadableStream 接收 | AC-09 |
