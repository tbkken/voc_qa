## 根因

`QaPipeline.ask()`（`app/pipeline.py`）捕获 DuckDB 执行异常后直接返回错误，
没有将错误信息反馈给 LLM 进行修正。

LLM 生成的 SQL 可能存在语义错误（如 GROUP BY 遗漏、列名误用、函数不正确），
这类错误通过了 sql_guard（只检查安全性），但 DuckDB 执行时抛出异常。
典型案例：
```
Binder Error: column "pt_d" must appear in the GROUP BY clause
or must be part of an aggregate function.
```

SPEC-001 无 AC 覆盖"SQL 执行失败 → LLM 修正重试"场景，属于 Spec 遗漏。

## 修复内容

**`app/llm.py` 新增 `fix_sql(question, bad_sql, error_msg)`：**
- 以多轮对话形式构建 prompt：system=schema prompt，user=原始问题，
  assistant=出错 SQL，user=错误信息 + 修正要求
- LLM 返回修正后的 SQL

**`app/pipeline.py` 执行块改为重试循环：**
- `for attempt in range(2)` 最多尝试 2 次
- 首次失败：调用 `fix_sql` 获取修正 SQL → `guard.check` 校验 → 若通过则继续循环重试
- 修正 SQL 未通过校验 / 重试仍失败：退出循环，返回最终错误
- 成功执行后，后续步骤（narrate、suggest_chart、AskResult）统一使用 `exec_sql`

**新增 SPEC-001 AC-26**

## 影响文件

- `app/llm.py`（新增 `fix_sql`）
- `app/pipeline.py`（执行块加重试循环，统一用 `exec_sql`）
- `docs/spec/SPEC-001-insight-analysis.md`（AC-26）
