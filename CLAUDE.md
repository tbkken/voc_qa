# CLAUDE.md — 项目规范与行为约束

## 新需求处理流程（强制）

**收到新功能需求时，禁止直接写代码。必须先走以下流程：**

1. **查 Spec**：读取 `docs/spec/` 下相关 Spec，确认需求是否已有对应 AC
   - 已有 AC → 直接进入步骤 3
   - 无对应 AC → 进入步骤 2
2. **补写 Spec**：调用 `spec-writer` skill，将需求转化为带 AC 的 Spec，**输出后暂停等待用户确认**
3. **拆任务**：调用 `ticket-mgr` skill，将 AC 拆分为可执行任务
4. **用户确认任务列表后**，方可开始实现

**禁止跳过 Spec 直接实现，即使需求看起来很简单。**

---

## Bug 处理流程（强制）

**遇到 bug 时，必须先走两阶段流程，不得直接修代码。**

### 阶段一：分析（自动执行，输出后暂停）

1. 读取 `docs/bugs/INDEX.md`，分配下一个编号
2. 定位根因（文件 + 函数 + 行号 + 原因）
3. 查 Spec，判断分流：A（代码写错）/ B（Spec 遗漏）/ C（需求歧义）/ D（无 Spec）
4. 输出分析报告，列出可选动作 A/B/C/D，**暂停等待用户决策**

### 阶段二：执行（按用户选择的字母组合）

- **[A]** 补充 / 更新 Spec AC
- **[B]** 修复代码
- **[C]** 补充测试
- **[D]** 生成 bug 记录（.md + .json + 更新 INDEX.md）

**禁止跳过阶段一直接修代码，即使根因已经明确。**

## Spec 变更规范

- 新增 AC 编号续接当前最大编号
- 同步更新：US 下的 AC 列表、数据模型表（如涉及）、AC 汇总表
- Bug 修复新增的 AC 末尾标注：`[已修复] YYYY-MM-DD BUG-XXX，由自测新增`

## Bug 记录规范

- 路径：`docs/bugs/BUG-NNN-YYYY-MM-DD-标题.md` + 同名 `.json`
- 每次生成记录后必须更新 `docs/bugs/INDEX.md`
- JSON 中 `resolution` 枚举：`full` / `code_only` / `ignored` / `pending`

## Git 提交规范（强制）

**每次完成以下任一动作后，必须立即做一次 git commit 到主分支（main）：**

1. **Bug 修复完成**（阶段二执行完毕，服务重启验证无 ERROR）
2. **Task / 功能实现完成**（前后端均可用，服务验证无 ERROR）
3. **Spec 新增 / 变更**（可与对应 Bug 修复或功能实现合并提交）

### Commit message 格式

```
fix: BUG-XXX 标题简述
feat: 功能/Task 标题简述
docs: SPEC-001 AC-XXX 描述
```

### 规则

- 提交前必须确认服务重启后无 ERROR，再执行 git commit
- 直接提交到 main 分支，不创建 feature branch，提交后立即 `git push origin main`
- 每次提交范围对应一个原子变更（一个 bug 或一个 task），不批量堆积
- 若一次实现同时涉及代码 + Spec + bug 记录，合并为一个 commit

## Commands

```bash
# Install dependencies
pip install -r requirements.txt

# Generate schema config from a CSV file (required before first run)
python data/init.py data/sample_voc.csv

./run.sh

# Start manually with auto-reload
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

# Run all tests (pipeline tests skipped if LLM not configured)
```

## Architecture

This is a Voice of Customer (VoC) Q&A system: users ask questions in Chinese → LLM generates SQL → DuckDB executes → LLM narrates results → response includes chart/table hints.

**Core data flow** (`app/pipeline.py`):
1. `llm.generate_sql(question)` — builds prompt from `schema_config.json` (enums, few-shots, field semantics), returns SQL in a fenced block
2. `sql_guard.check(sql)` — 6-layer validation: strips comments, rejects non-SELECT, enforces table whitelist (`fact_voc` only), blocks multi-statements, auto-appends `LIMIT 1000`
3. `engine.execute(sql)` — DuckDB in-memory query, returns `{columns, rows, row_count, truncated, elapsed_ms}`
4. `llm.narrate_result(question, sql, data)` — interprets results in ~150-char Chinese natural language
5. `_suggest_chart(sql, data)` — returns hint: `metric`, `line`, `bar`, or `table`

**Module responsibilities:**
- `app/engine.py` — DuckDB wrapper; `load_csv()` populates `fact_voc`, `execute()` runs queries
- `app/llm.py` — OpenAI-compatible HTTP client (httpx); `build_sql_system_prompt()` dynamically includes real enum values from config
- `app/sql_guard.py` — all SQL security; edit here to adjust allowed tables or forbidden keywords
- `app/config.py` — cached loader for `data/schema_config.json`; call `reload_schema_config()` after CSV upload
- `app/main.py` — FastAPI app with 8 routes; global `engine`, `llm`, `pipeline` singletons initialized at startup
- `data/init.py` — scans a CSV with DuckDB and writes `data/schema_config.json` (enums, few-shots, date ranges, field descriptions); this config drives all LLM prompts

**`data/schema_config.json`** is the central artifact — it bridges real CSV data to LLM prompts. It is auto-generated by `data/init.py` and must exist before the server starts. After uploading a new CSV via `POST /api/upload`, `init.py` is re-run automatically and the config is reloaded.

**Frontend** (`web/index.html`) — single-page chat UI (vanilla JS, dark theme, ECharts for charts). Communicates with the backend via `fetch()` to `/api/*` endpoints.

**Table schema** (`fact_voc`): 11 VARCHAR columns — `pt_d`, `source_sound_id`, `comment_time`, `business_category_name`, `content`, `is_show`, `data_channel`, `emotion`, `first_category`, `fifth_category`, `keywords_emotion`.

## Key Design Constraints

- All data is in-memory (DuckDB `:memory:`); no external databases.
- `SqlGuard` only permits queries against the `fact_voc` table. Adding a new table requires updating the whitelist in `app/sql_guard.py`.
- LLM prompts are built dynamically from `schema_config.json` — never hardcode field names or enum values; always derive them from the config.
- Tests set `LLM_MOCK=1` automatically; mock LLM uses keyword matching (defined in `app/llm.py`), not a real API call.
